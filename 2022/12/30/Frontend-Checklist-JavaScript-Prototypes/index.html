JavaScript Checklist: Prototypes

Gloria Su
2022-12-30

1. 原型设计模式与规则

  • 原型模式：使用构造函数的 prototype 属性来指定那些应该共用的属性和方法。
  • 创建对象：new 一个构造函数；字面量 {} 创建；隐式的装箱操作和 Object 的一些方法。
  • 原型规则：
    1. 构造函数的 prototype 指向共享的对象，通过 new 构造函数来获取这些共享的属性和方法。
    2. new 形成的实例，就是把__proto__指向来构造函数的 prototype。
    3. 每个对象都有一个__proto__属性指向一个 Object。
    4. 当试图得到一个对象的某个属性时，如果这个对象本身没有这个属性，那么会去它的__proto__(即它的构造函数的prototype)中去寻找。

2. 手写 instanceof

  • instanceof 是用来判断引用类型，也可以通过原型链实现继承关系的判断。
  • instanceof 的实现实际上是调用 JS 内部函数 [[HasInstance]] 来实现的。
  • 实现原理：就是只要右边变量的 prototype 在左边变量的原型链上即可。因此，instanceof 在查找的过程中会遍历左边变量的原型链，直到找到右边变量的 prototype，如果查找失败，则会返回 false。

4. 实现继承的方法 优缺点

  1. 原型链继承：利用原型让一个引用类型继承另一个引用类型的方法和属性。
  • 优点：
    1. 父类新增原型方法/原型属性，子类都能访问到。
    2. 实例是子类的实例，也是父类的实例。
  • 缺点：
    1. 来自原型对象的引用属性是所有实例共享的，原型上属性的改变会作用到所有的实例上。
    2. 创建子类实例时，无法向父类构造函数传参。
    3. 无法实现多继承。
  1. 构造函数继承：在构造子类构造函数时内部使用 call 或 apply 来调用父类的构造函数。
  • 优点：
    1. 实现了属性的私有化。
    2. 创建子类实例时，可以向父类传递参数。
    3. 可以实现多继承（call多个父类对象）。
  • 缺点：
    1. 实例并不是父类的实例，只是子类的实例。
    2. 只能继承父类的实例属性和方法，不能继承原型属性/方法。
    3. 无法实现函数复用，每个子类都有父类实例函数的副本，影响性能。
  1. 实例继承：在子类中 new 父类。
  • 优点：
    1. 不限制调用方式，不管是 new 子类()还是子类()，返回的对象具有相同的效果。
  • 缺点：
    1. 实例是父类的实例，不是子类的实例。
    2. 不支持多继承。
  1. 组合继承：将原型链和借用构造函数的技术组合到一块。instanceof 和 isPrototypeOf 也能够用于识别基于组合继承创建的对象。
  • 优点：
    1. 可以继承实例属性/方法，也可以继承原型属性/方法。
    2. 既是子类的实例，也是父类的实例。
    3. 不存在引用属性共享问题。
    4. 可传参。函数可复用。
  • 缺点：
    1. 调用了两次父类构造函数，生成了两份实例（子类实例将子类原型上的那份屏蔽了）。
  1. 寄生继承：将 sub.prototype = new super 改为 sub.prototype = Object.create (supper.prototype)，避免了组合继承中构造函数调用了两次的弊端。
  • 优点：
    1. 堪称完美。
  • 缺点：
    1. 实现较为复杂。
  1. Class 继承：
  • 优点：
    1. 堪称完美。
  • 缺点：
    1. es6 语法糖,需要一定的浏览器兼容性或者 polyfill。
